## Technical Context

### Technology Stack

- **Backend**: Laravel 12, PHP 8.4, Pest v4
- **Event Sourcing**: Spatie laravel-event-sourcing v7 (existing domain events as triggers)
- **Business Logic**: Lorisleiva Actions (`AsAction` trait)
- **Database**: SQLite (dev), schema-only migrations
- **Frontend**: Next.js 16, React 19, TypeScript, TanStack Query v5, Zustand v5
- **Styling**: Tailwind CSS v3, shadcn/ui
- **Forms**: React Hook Form + Zod

### Dependencies

**Internal (must exist before 024-NTF ships)**:
- `JournalEntrySubmitted`, `JournalEntryApproved`, `JournalEntryRejected` events — ✅ exist
- `InvoiceSent`, `InvoicePaymentReceived` events — ✅ exist
- `InvoiceOverdue` signal — generated by a scheduled command (needs adding to existing overdue check or new command)
- `SyncBankFeed` action — ✅ exists; must fire notification hooks on completion/failure
- `JobShareToken` model + `JobShareLinkViewed` event — ⚠️ 022-CPV not built on this branch; plan treats this as a stub and adds a placeholder listener that activates when CPV ships
- Event payloads for `InvoiceSent` and `InvoicePaymentReceived` are missing the `sentBy` / `invoiceId` user identifiers — events will need enrichment (see Phase 1)

**External**:
- None — no third-party push service; polling only

### Constraints

- Polling interval: 30 seconds, using TanStack Query's `refetchInterval`
- Unread count endpoint must respond in < 200 ms for workspaces with up to 10,000 records
- Activity feed first-page load < 1 second for up to 1,000 records
- No WebSocket (Reverb) in this epic — architecture must be Reverb-ready (see FR-023)
- Notifications are immutable after creation — read/dismissed state only
- No notification preferences UI in this epic (deferred)

## Design Decisions

### Data Model

#### `notifications` table (tenant-scoped)

```
id                  bigint unsigned PK auto-increment
uuid                char(36) UNIQUE NOT NULL
workspace_id        bigint unsigned NOT NULL FK workspaces
user_id             bigint unsigned NOT NULL FK users
type                varchar(50) NOT NULL  -- NotificationType enum value
title               varchar(255) NOT NULL
body                text NOT NULL
subject_type        varchar(100) NULLABLE  -- morph: App\Models\Tenant\Invoice etc.
subject_id          bigint unsigned NULLABLE
actor_id            bigint unsigned NULLABLE FK users
read_at             timestamp NULLABLE
dismissed_at        timestamp NULLABLE
created_at          timestamp NOT NULL
updated_at          timestamp NOT NULL
```

**Indexes**:
- `(workspace_id, user_id, read_at)` — unread count query
- `(workspace_id, user_id, dismissed_at, created_at)` — feed pagination
- `(subject_type, subject_id)` — deep-link resolution
- `(type, workspace_id)` — type filter on feed

#### `NotificationType` enum

```php
enum NotificationType: string {
    case JeSubmitted        = 'je_submitted';
    case JeApproved         = 'je_approved';
    case JeRejected         = 'je_rejected';
    case InvoiceSent        = 'invoice_sent';
    case InvoicePaid        = 'invoice_paid';
    case InvoiceOverdue     = 'invoice_overdue';
    case JobShareViewed     = 'job_share_viewed';
    case BankFeedSynced     = 'bank_feed_synced';
    case BankFeedError      = 'bank_feed_error';
    case ReconciliationCompleted = 'reconciliation_completed'; // stub, not wired
}
```

Each case has: `label()`, `icon()`, `filterCategory()` → one of: Approvals, Invoicing, Banking, Jobs, All.

#### `Notification` Model

- Extends standard Eloquent (NOT event-sourced — notifications are ephemeral operational records, not financial facts)
- `HasUuids` trait for URL-safe identifiers
- `tenant_id` scoping via `workspace_id` global scope
- Relationships: `user()`, `workspace()`, `actor()`, `subject()` (morphTo)
- Scopes: `unread()`, `active()` (not dismissed), `forWorkspace()`
- No aggregate root — notifications are not financial events

### Listener Architecture

Rather than projectors (event-sourcing read models), notifications use standard Laravel **event listeners** registered in `EventServiceProvider`. Each stored event fires through the Spatie projector pipeline, but we hook standard `Observer`-style listeners for notification creation.

**Pattern**: Each domain event gets a dedicated listener:

```
app/Listeners/Notifications/
├── NotifyApproversOnJeSubmitted.php
├── NotifyBookkeeperOnJeApproved.php
├── NotifyBookkeeperOnJeRejected.php
├── NotifyInvoiceSender.php          (InvoiceSent)
├── NotifyOwnersOnInvoicePaid.php    (InvoicePaymentReceived)
├── NotifyOwnersOnBankFeedSynced.php
├── NotifyOwnersOnBankFeedError.php
└── NotifyOnJobShareViewed.php       (stub — CPV dependency)
```

Overdue invoice notifications are fired from a **scheduled command** (`SendInvoiceOverdueNotifications`) that runs nightly, similar to 023-EML's overdue reminders.

**Important**: Spatie event-sourcing stored events are dispatched as standard Laravel events too. Listeners receive the raw stored event object — access aggregate UUID via `$event->aggregateRootUuid()`.

### API Contracts

All routes under `/api/v1/` within the workspace middleware group (`SetWorkspaceContext`):

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/notifications/unread-count` | Returns `{ count: N }` — polling endpoint |
| GET | `/notifications` | Paginated feed (25/page), optional `?type=` filter |
| GET | `/notifications/recent` | Top 10 for bell dropdown |
| POST | `/notifications/mark-all-read` | Marks all workspace notifications read |
| PATCH | `/notifications/{uuid}/read` | Marks one notification read |
| DELETE | `/notifications/{uuid}` | Dismisses (soft-deletes) one notification |

`NotificationResource`:
```json
{
  "uuid": "...",
  "type": "je_approved",
  "title": "Journal entry approved",
  "body": "JE-042 approved by Sarah Chen",
  "icon": "check-circle",
  "filter_category": "Approvals",
  "subject_url": "/journal-entries/abc-123",
  "actor": { "name": "Sarah Chen" },
  "read_at": null,
  "created_at": "2026-03-15T10:30:00Z",
  "relative_time": "3 minutes ago"
}
```

frontend/src/

## Implementation Phases

### Phase 1 — Backend Foundation

**Goal**: Notification model, migration, enum, API endpoints, and `CreateNotification` action.

Tasks:
1. `NotificationType` enum (`app/Enums/NotificationType.php`) with `label()`, `icon()`, `filterCategory()` methods
2. Migration: `create_notifications_table` with all columns and indexes
3. `Notification` model with `HasUuids`, scopes, morph `subject()` relationship, registered in morph map
4. `CreateNotification` action — single entry point for all notification creation; idempotent
5. `NotificationController` — 6 endpoints (unread-count, index, recent, mark-all-read, patch-read, delete)
6. `NotificationResource` + `NotificationCollection`
7. API routes registered in `routes/api.php`
8. Feature tests for all 6 endpoints (auth, workspace scoping, response shape)

**Event payload enrichment** (required before listeners work):
- `InvoiceSent` — add `sentBy: int` (user ID of sender) to payload and `InvoiceAggregate::markSent()`
- `InvoicePaymentReceived` — already has `recordedBy`; aggregate UUID available via listener

### Phase 2 — Event Listeners

**Goal**: Wire all notification triggers to existing domain events.

Tasks:
1. `NotifyApproversOnJeSubmitted` — queries workspace users with `approver` role; excludes submitter; calls `CreateNotification` per recipient
2. `NotifyBookkeeperOnJeApproved` — notifies `submittedBy` user from JE record
3. `NotifyBookkeeperOnJeRejected` — notifies `submittedBy`, includes rejection reason from event payload
4. `NotifyInvoiceSender` — on `InvoiceSent`, notifies `sentBy` user with invoice number + contact name
5. `NotifyOwnersOnInvoicePaid` — on `InvoicePaymentReceived`, notifies all owners + accountants + bookkeeper raiser (de-duped); uses `$event->aggregateRootUuid()` to resolve invoice
6. `SendInvoiceOverdueNotifications` command — nightly scheduled command; marks `overdue_notification_sent_at` on invoice to prevent repeats; notifies owners + accountants
7. `NotifyOwnersOnBankFeedSynced` — hook into `SyncBankFeed::handle()` post-import; fires on success
8. `NotifyOwnersOnBankFeedError` — wraps `SyncBankFeed::handle()` in try/catch at the controller/job level
9. `NotifyOnJobShareViewed` stub — listener registered but returns early if `JobShareToken` model not found (CPV dependency)
10. Register all listeners in `EventServiceProvider`
11. Feature tests for each listener (seed event, assert notification records)

**Deduplication for overdue**: Add `overdue_notified_at` nullable timestamp column to `invoices` table.
**Deduplication for job_share_viewed**: Store `notified_at` per token (deferred to CPV epic — stub only).

### Phase 3 — Frontend

**Goal**: Bell, dropdown, activity feed page, polling.

Tasks:
1. `NotificationType` constants in `frontend/src/types/notifications.ts`
2. `use-unread-count.ts` — TanStack Query with `refetchInterval: 30_000`, stale on workspace switch
3. `use-recent-notifications.ts` — fetches `/notifications/recent`, triggered on bell open
4. `use-notifications.ts` — paginated feed with `type` filter param, infinite query or page-based
5. `notification.ts` Zustand store — bell open state, optimistic mark-read
6. `notification-item.tsx` — renders icon, title, body excerpt, relative time, read indicator, deep link
7. `notification-bell.tsx` — bell icon with badge (99+), keyboard shortcut `Cmd+/` opens AI panel (bell uses click only — no conflicting shortcut)
8. `notification-dropdown.tsx` — popover with top 10, "Mark all read" button, "View all" link, auto-mark-read on open
9. `notification-type-filter.tsx` — tab strip: All / Approvals / Invoicing / Banking / Jobs
10. `notification-feed.tsx` — paginated list with dismiss button per row
11. `/notifications/page.tsx` — activity feed page with filter, pagination, "Mark all as read", empty state
12. Wire bell into `layout.tsx` header
13. Add `G then N` keyboard shortcut to navigate to `/notifications` (update `CLAUDE.md` shortcuts table)

### Phase 4 — Scheduled Jobs & Purge

Tasks:
1. Schedule `SendInvoiceOverdueNotifications` daily at 06:00 UTC in `Console/Kernel.php` (or `routes/console.php` if using Laravel 12 style)
2. `PurgeOldNotifications` command — deletes `dismissed_at IS NOT NULL OR created_at < 90 days ago`; scheduled daily at 03:00 UTC
3. Feature tests for both commands

### Phase 5 — Browser Tests

Tasks:
1. `NotificationsTest.php` — Pest/Playwright browser tests:
   - Bell badge appears when unread notifications exist
   - Dropdown opens showing notifications
   - Auto-mark-read on open clears badge
   - "View all" navigates to `/notifications`
   - Activity feed renders with filter tabs
   - Dismiss removes notification from feed
   - "Mark all as read" clears badge

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| `InvoiceSent` event missing `sentBy` payload | High | High | Enrich event payload in Phase 1 — breaking change caught by existing tests |
| `JobShareToken` CPV dependency not built | High | Low | Stub listener, documented dependency, activates when CPV ships |
| Polling 30s adds DB load at scale | Medium | Medium | Indexed `(workspace_id, user_id, read_at)` — COUNT query is fast |
| Listener failures blocking domain operations | Low | High | All listeners queued or wrapped in try/catch — failures logged, not re-thrown |
| Duplicate notifications on bulk imports | Medium | Medium | Dedup guards per listener type (overdue once, bank feed per-sync) |
