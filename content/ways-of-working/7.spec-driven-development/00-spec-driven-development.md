## The Problem

AI is powerful. AI is also dangerous.

Left unchecked, AI will confidently implement the wrong thing. It hallucinates. It fills gaps with assumptions. It builds exactly what you asked for—which isn't always what you meant. And it does all of this at terrifying speed.

The result? Teams ship broken features faster than ever before.

---

## What Goes Wrong

| Without Structure | What Happens |
|-------------------|--------------|
| Vague prompts | AI fills gaps with hallucinations |
| No verification | Wrong code ships to production |
| No documentation | Nobody knows why it was built this way |
| No human review | AI mistakes compound undetected |
| No acceptance criteria | "Done" means nothing |

AI doesn't need less oversight. It needs *more*.

---

## The Solution: Human-in-the-Loop

Spec-driven development puts humans back in control:

1. **Define before building** — The spec is written by humans, not generated by AI
2. **Review at every gate** — Human gatekeepers approve before work proceeds
3. **Test against the spec** — AI output is verified against human-defined criteria
4. **Document the intent** — The spec captures *why*, not just *what*

AI becomes a tool that implements human decisions, not a replacement for human thinking.

**The spec is the source of truth. AI is just the implementation engine.**

---

## The Core Idea

Traditional development: **Think → Code → Document**

Spec-driven development: **Specify → Clarify → Visualize → Plan → Implement**

The spec becomes the contract between what you want and what AI builds. Quality gates ensure nothing moves forward until it's ready.

## Scaling to Complexity

Not every feature needs the full journey. Scale to the work:

| Feature Size | What You Need |
|--------------|---------------|
| **Trivial** (typo fix) | Just implement it |
| **Small** (single component) | Spec → Plan → Implement |
| **Medium** (feature) | Full workflow, simplified gates |
| **Large** (epic) | Full workflow, all gates, multiple iterations |

The gates exist to catch problems early. For trivial work, the cost of the gates exceeds the risk. For large work, skipping gates creates expensive rework.

## What Makes a Good Spec

The quality of your spec determines the quality of AI's implementation.

| Vague | Precise |
|-------|---------|
| "Users can manage notifications" | "Users can toggle email vs in-app for each notification type" |
| "Fast loading" | "Dashboard loads within 2 seconds on 3G" |
| "Better UX" | "Reduce clicks from 5 to 2 for common actions" |

A good spec is:
- **Specific** — No ambiguity for AI to misinterpret
- **Complete** — Happy path, edge cases, error states
- **Testable** — Every statement is verifiable
- **Living** — Updated when implementation reveals gaps

## Getting Started

1. **Start small** — Pick a minor feature to practice
2. **Run `/trilogy-idea`** — Capture the problem
3. **Run `/speckit-specify`** — Detail the solution
4. **Run `/trilogy-clarify`** — Fill gaps through different lenses
5. **Run `/trilogy-design`** — Visualize what users will see
6. **Run `/speckit-plan`** — Plan the technical approach
7. **Run `/speckit-implement`** — Let AI build it

Watch how the quality of your spec directly affects the quality of the implementation.
